# Инструкции для код-агента (Codex / VS Code)

Цель: быстрые и безопасные итерации. Любая правка — минимальный diff, измеримая приемка, без “попутных улучшений”.

## 1) Рабочий цикл
- 1 задача = 1 ветка = 1 PR.
- Запрещено смешивать: auth + БД + UI в одном PR (кроме минимального “сквозного” подключения).
- Каждый PR обязан содержать: описание изменений, чек-лист ручных тестов, откат (rollback).

## 2) Источник правды
- Требования по текущему PlanKanban: `docs/PLANKANBAN_DOCS.md`.
- ТЗ деплоя через VPS/Worker: `docs/KANBAN_DEPLOY_TZ.md`.
- Архитектура и контракты: `docs/ARCHITECTURE.md`, `docs/API.md`, `docs/DATABASE.md`.

## 3) Инварианты (нельзя сломать)
- Kanban UI должен открываться и работать без LLM (LLM — опционально).
- Ключи провайдеров LLM никогда не попадают в браузер и git.
- Любые критичные ошибки должны быть видимы пользователю (fail-state), не “silent”.

## 4) Правила изменения фронта
- Минимизировать изменения в `apps/web/index.html` до момента, пока не появится стабильный backend API.
- Любое изменение UI: снапшот до/после в PR (скриншоты или GIF).

## 5) Правила изменения бэкенда
- Все endpoints документировать в `docs/API.md`.
- Все миграции/схема: в `docs/DATABASE.md` + SQL миграции.
- Логи: без секретов и без полного payload LLM.

## 6) Тестирование (минимум)
- Auth: login/logout, запрет без сессии.
- RBAC: сотрудник видит только свои задачи, техлид — постановка/редактирование, админ — всё.
- Tasks: CRUD, фильтры, сортировка, перемещение по колонкам.
- LLM: запрос проходит через gateway; ошибки провайдера корректно отображаются.
