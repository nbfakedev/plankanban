5. Зоны ответственности модулей
5.1 Backend
Проекты: создание, редактирование, активация, возврат активного проекта.
Задачи: CRUD, перемещение между колонками, reorder внутри колонки, отдача задач по проекту.
Светофор задач: агрегаты backlog / in work / done.
Бюджет: расчет earned / total / progress.
Таймер проекта: серверный runtime и customer delay.
Аудит: запись task_events атомарно в одной транзакции с изменением задач.
Аналитика: расчет project / task / time / budget metrics.
LLM-заглушки и будущие точки входа: import/excel и llm/task-dialog.
5.2 Frontend
Отображение Kanban-доски и колонок.
Переключение проекта без reload страницы.
Рендер светофора, бюджета и таймера в header-панели.
Live-поиск и stage-фильтры.
DnD и persist reorder через API.
Profile-панель с аналитикой.
5.3 База данных
Хранение задач, проектов, истории действий и таймера.
Единый источник правды для текущего состояния PlanKanban24.
Основа для аналитики и будущих LLM-агентов.

7. Бизнес-логика PlanKanban24
7.1 Kanban-логика
Колонки PlanKanban24: backlog, todo, doing, review, done.
Перемещение задачи между колонками меняет поле tasks.col.
Порядок задач внутри колонки хранится в tasks.position.
Этап задачи хранится в tasks.stage и используется в stage-фильтрах.
Поиск работает по title, descript, notes, stage, agent, task.id, public_id и UUID.

7.3 Таймер проекта
Таймер PlanKanban24 полностью серверный. Он не должен зависеть от состояния вкладки, браузера или сети.
POST /timer/start запускает или возобновляет project runtime.
POST /timer/stop останавливает project runtime и запускает customer delay timer.
GET /timer возвращает текущее вычисленное состояние: project_time, client_delay_time, deadline, status и технические *_ms-поля для фронта.
Customer delay не обнуляется и накапливается как отдельная часть хронологии проекта.

11. Отдельный блок: LLM в PlanKanban24
Этот раздел фиксируется отдельно, потому что LLM-слой в PlanKanban24 еще не подключен физически, но архитектурно уже предусмотрен. Codex должен отдельно сверить его с кодом и дополнить по факту реализации.
11.1 Цели LLM-слоя
Импорт Excel со списком задач и автоматическим разбором колонок, этапов, описаний и релизов.
Создание новой задачи через диалог с LLM: уточнение, нормализация, предложение финальной формы задачи.
Дальнейшая работа AI-агентов с Kanban на базе API PlanKanban24.
11.2 Предусмотренные точки входа

11.3 Целевая схема LLM-подключения
User / Admin   │   ├── upload Excel   │   └── dialog request                │                ▼         PlanKanban24 API                │                ├── validates request / file                ├── prepares normalized prompt                ├── sends request to LLM provider                ├── receives structured response                └── writes tasks into PostgreSQL
11.4 Требования к LLM-слою
LLM не должен иметь прямой доступ к БД PlanKanban24.
Все операции записи должны выполняться только через backend API или внутренний сервисный слой сервера.
Результат LLM должен приводиться к структурированному объекту задачи, а не к свободному тексту.
Для импорта Excel должен сохраняться журнал операции: кто загрузил файл, какой проект был активен, сколько задач создано, какие записи не удалось распарсить.
Для task-dialog должен быть этап подтверждения администратором перед записью задач в PlanKanban24.
11.5 Что Codex должен проверить и дополнить по LLM
Фактическое наличие endpoint-ов /import/excel и /llm/task-dialog в коде.
Наличие или отсутствие provider-слоя (Anthropic / OpenAI / worker bridge / stub mode).
Формат входных payload и формат ответа.
Схему хранения логов LLM-операций, если такая таблица уже существует.
Проверку безопасности: ключи, env-переменные, отсутствие утечек промптов и секретов в логах.
12. Отдельный блок: подключение PlanKanban24 к сторонним сервисам
Этот раздел фиксирует не только AI-агентов, но и любой внешний сервис, который будет работать с PlanKanban24: BI, автоматизация, интеграционные мосты, внешние оркестраторы и LLM-платформы.
12.1 Базовый принцип интеграции
Сторонние сервисы подключаются к PlanKanban24 только через API. Прямое чтение/запись в PostgreSQL запрещены.
12.2 Типы внешних сервисов
AI-агенты (Planner, Executor, QA, Analyst).
BI / аналитические панели.
Интеграции с файлами и импортом (Excel, CSV).
Оркестраторы автоматизации и workflow engines.
Внешние инструменты управления проектами, если появится мост синхронизации.

12.4 Минимальные требования безопасности для внешних сервисов
Каждый внешний сервис должен работать под отдельным типом учетной записи или сервисным токеном.
Права должны быть ограничены по scope: read-only, tasks-write, metrics-read и т.д.
Все внешние действия должны попадать в audit trail PlanKanban24.
Нельзя давать внешним сервисам доступ к env-файлам, миграциям и внутренним SQL-командам.
12.5 Что Codex должен дополнить по внешним интеграциям
Проверить, существует ли уже service-account модель или отдельный auth-слой для агентов.
Проверить, есть ли rate limit / idempotency / безопасная схема для внешних write-операций.
Зафиксировать, какие endpoint-ы уже пригодны для agent-to-kanban взаимодействия без доработки.
Проверить, есть ли webhooks / polling endpoints для внешнего контура.

13.4 Ограничения для агентов
Агент не должен перескакивать через API и писать напрямую в PostgreSQL.

Агент не должен массово удалять задачи без отдельного подтвержденного workflow.
LLM-агенты должны оставлять после себя трассируемый след действий.
14. Спецификация протокола работы агентов с Kanban
Ниже фиксируется рекомендованный рабочий протокол multi-agent взаимодействия с PlanKanban24. Это именно протокол поведения, а не только список endpoint-ов.
14.1 Общий протокол
Агент определяет активный проект или получает project_id извне.
Агент загружает текущее состояние задач и метрик.
Агент анализирует контекст, не делая записи.
Агент формирует намерение действия: create, update, move, reorder, analyze.
Агент вызывает API PlanKanban24.
После вызова агент проверяет результат повторным чтением данных или истории действий.
Если действие критичное, агент обязан ждать подтверждение человека или управляющего агента.
14.2 Протокол для Planner Agent
Читает задачи проекта и stages.
Создает новые задачи через POST /projects/:projectId/tasks.
Не перемещает существующие задачи без отдельного правила.
14.3 Протокол для Executor Agent
Читает assigned / visible задачи.
Меняет статус и поля задачи через PATCH /tasks/:id и POST /tasks/:id/move.
Не удаляет задачи.
14.4 Протокол для QA Agent
Читает задачи в review / doing.
Проверяет историю и метрики.
Перемещает задачу в review или done только по правилам проекта.
14.5 Протокол для Analyst Agent
Читает только GET endpoints.
Использует /projects/:projectId/metrics, /stats/tasks, /stats/budget, /timer, /projects/:projectId/events.
Не делает write-операций.
15. Архитектура multi-agent системы поверх PlanKanban24
PlanKanban24 выступает как системный backend и источник правды. Multi-agent слой строится поверх него, но не внутри его БД.

15.2 Принцип безопасной оркестрации
Coordinator принимает решения, но не должен обходить ограничения API.
Каждый агент должен быть ограничен scope-ами.
Все write-действия агентов должны попадать в audit trail.
Критичные изменения (удаление, массовая перестройка, изменение project settings) должны требовать подтверждения человека.

17. Текущее заключение
PlanKanban24 в текущем состоянии уже является полноценной backend-платформой для Kanban-управления: ядро доски, аудит, таймер и аналитика готовы.
Главные недостроенные контуры — это LLM и внешняя агентная интеграция. Поэтому именно блоки «LLM» и «подключение сторонних сервисов» выделены отдельно и должны быть следующей зоной детализации.
Этот документ следует считать базовой технической спецификацией PlanKanban24. После передачи Codex он должен быть дополнен прямыми ссылками на код, фактическими контрактами API и всеми деталями, которые можно подтвердить только в репозитории.
