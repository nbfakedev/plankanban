# PlanKanban — Документация проекта
> Для передачи на детальную разработку. Версия: март 2026.

---

## 1. Обзор проекта

**PlanKanban** — одностраничное веб-приложение (SPA) для управления задачами разработки. Работает полностью в браузере без бэкенда. Изначально создан для проекта mossb.ru, но поддерживает несколько проектов.

**Ключевые принципы:**
- Zero-backend: весь стейт в `localStorage`
- Single HTML file: CSS + JS + HTML в одном файле
- AI-first: Claude встроен в три сценария создания и работы с задачами
- Dark/Light тема из коробки
- Drag-and-drop между колонками

**Открывается:** кнопка "＋ Задача" или "＋ задача" в футере колонки

3. Claude уточняет детали в диалоге
4. Когда задача ясна — Claude генерирует `TASK_JSON::{ ... }` в конце ответа
5. Парсится regex: `/TASK_JSON::(\{[\s\S]*?\})\s*$/`

7. Кнопки: **✓ Добавить в Backlog** или **↩ Уточнить**
8. При подтверждении: генерируется ID (инкремент по этапу), задача добавляется в `tasks[]`, сохраняется

## 9. Система таймеров

Три состояния: `stopped` → `project` → `delay` → `project` → ...

```
[▶ Старт]  →  Запускается project timer
[⏸ Стоп]  →  Останавливается project timer, запускается delay timer (красный chip появляется)
[▶ Старт]  →  Останавливается delay timer, возобновляется project timer
```

**Персистентность между сессиями:**
При закрытии страницы сохраняется `mossb_timer_wall = Date.now()`. При открытии вычисляется elapsed = `Date.now() - timerWall` и прибавляется к соответствующему таймеру.

**Отображение project timer:**
- Недели / день недели (0-6) / HH:MM
- Цветовая индикация недель: >65% срока → `warn` (жёлтый), >85% → `danger` (красный)

**Delay timer (таймер задержки клиента):** красный chip с теми же полями. Показывает время простоя по вине клиента.

## 11. Drag & Drop

Нативный HTML5 Drag and Drop API.

**Флоу:**
1. `dragstart` → сохраняется `dragId`, `dragSourceCol`; карточка получает класс `dragging` (opacity 0.45, rotate 1.5deg, scale 1.03)
2. `dragover` → `preventDefault()`, отрисовывается `card-drop-indicator` (золотая линия между картами)
3. `dragenter/dragleave` → highlight колонки (`drag-over-col`, `drag-over`)
4. `drop` → обновляется `t.col`, карточка переставляется в DOM без полного ре-рендера
5. `dragend` → очистка всех hover-классов, сброс состояния через 50ms (чтобы click-handler не срабатывал)

**Позиция дропа** определяется через `getDragAfterEl()`: находит ближайшую карточку ниже курсора.

## 12. Поиск и фильтрация

**Поиск** (real-time, `oninput`): фильтрует по `title`, `id`, `track`, `agent` (case-insensitive).

**Фильтр по этапу:** кнопки в stage-bar; `curStage = 'all' | 'A' | 'R1' | 'R1.1' | 'R2' | 'R3+' | 'F'`

Группировки в фильтре:
- `'R2'` → показывает задачи stage `R2` и `R2+`
- `'R3+'` → показывает задачи stage `R3+` и `R4+`

Поиск применяется поверх фильтра этапа.

## 13. Расчёт статистики

**Traffic light:** подсчёт задач по `col`
- Done = `col === 'done'`
- WIP = `col === 'inprogress'`
- Backlog = всё остальное

**Прогресс:** `done / tasks.length * 100%`

**Заработанный бюджет:**
```javascript
stages.forEach(s => {
  const stTasks = tasks.filter(t => t.stage === s);
  const donePct = stTasks.filter(t => t.col === 'done').length / stTasks.length;
  earned += Math.round((STAGE_BUDGETS[s] || 0) * donePct);
});
```
Логика: каждый этап вносит в "заработанное" пропорционально доле выполненных задач этапа.

### 14.6 Зачем Claude в канбане

1. **Создание задачи через диалог** — не форма с полями, а живой разговор с Tech Lead. Claude сам выбирает этап, агента, оценивает часы, пишет ТЗ. Снижает когнитивную нагрузку на архитектора.

2. **AI-чат внутри задачи** — контекстный ассистент знает стек, этап, агента, ТЗ задачи. Используется для декомпозиции на подзадачи, написания API-контракта, схемы БД, промпта для Cursor/Codex.

3. **Импорт через Claude** — Excel или текстовое описание автоматически превращается в структурированные задачи с правильными полями. Исключает ручной ввод при первичном заполнении бэклога.

## 18. Известные ограничения и что нужно доработать

### 18.1 Критичные
- **API ключ в клиенте** — Claude API key захардкожен/передаётся напрямую из браузера. Нужен прокси-эндпоинт.
- **localStorage лимит** — при большом количестве задач (~1000+) возможно переполнение (~5MB). Нужна миграция на IndexedDB или бэкенд.

### 18.2 Функциональные доработки под mossb.ru
- **Поле "Зависимости"** — сейчас строка, нужна визуализация граф-зависимостей
- **Фильтр по агенту** — нет фильтрации по `agent`
- **Экспорт** — нет выгрузки в Excel/CSV
- **Уведомления через n8n** — интеграция с Telegram при смене статуса задачи
- **Синхронизация** — задачи только локально, нет shared-доступа
- **Трек-фильтр** — фильтр по `track` (Backend, Frontend, DevOps...) отсутствует
- **Время работы на задаче** — нет трекинга времени по конкретной задаче
- **Архив** — нет архивирования выполненных задач

### 18.3 UX
- Нет подтверждения при удалении проекта
- Нет отмены удаления задачи (нет delete вообще — только через drag в архив)
- Нет пагинации/виртуализации колонок при 100+ задачах в одной колонке
